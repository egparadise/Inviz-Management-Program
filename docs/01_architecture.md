## 데이터 흐름 시나리오

### 시나리오 1: 회계 담당자가 새 매출 파일 업로드
1. 사용자 → `14.경영정보/26.결산자료/외상매출금 (2026-06).xlsx` 저장
2. 04:00 안전 동기화 자동 실행
3. `sync_core` → 파일 변경 감지 (mtime + SHA256)
4. 도메인 매칭: `re.compile("외상매출금")` → `sale_ar`
5. 같은 도메인 이전 파일과 비교: 이게 mtime 최신 → 선택
6. `handler_sale_ar` 실행: 이전 source_file 데이터 삭제 후 wide → long unpivot 적재
7. 무결성 검증: 매출 합계 +15% → OK
8. 벡터 DB 갱신: `sale_monthly` 청크 재인덱싱
9. 다음 날 사용자가 `/sales`에서 6월 매출 확인

### 시나리오 2: 사용자가 챗에서 "올해 KPI" 질문
1. 검색바 입력 → 패널 자동 열림
2. JS → `POST /chat/stream` (SSE)
3. `fast_intent_match("올해 KPI")`:
   - "올해" → year=2026
   - "KPI" → intent=kpi
   - 5ms 즉시 응답
4. `dispatch(intent="kpi", db)`:
   - 매출/매입/이익/차입금/계약 등 6개 지표 SQL 집계
5. 템플릿 요약 즉시 반환 (LLM 호출 없음)
6. 사용자에게 "2026년 매출 3.19억, 매입 1.43억, 이익 1.76억..." 표시

### 시나리오 3: 사용자가 "써밋영상의원 거래내역" 질문
1. fast match 키워드 매칭 실패 (거래처명 모름)
2. RAG 모드 활성화
3. LLM (llama3.1)로 의도 추출 → `search_party, party_name="써밋영상의원"`
4. `dispatch` → DB 결과 (Party + 매출 누계)
5. 동시에 벡터 검색 → 6개 청크 (써밋영상의원 [0.74], 2025-02 매출 등)
6. LLM에 컨텍스트 + 질문 → 토큰별 스트리밍 응답
7. 사용자에게 4개 단계 애니메이션 + 점점 늘어나는 답변 텍스트
